## 1. 文档导读
本文档是 `smart-gateway`的说明书，简要介绍了网关模块的架构设计与技术实现，也包含着我开发过程中的些许思考。以帮助其他学者更好地理解项目内容。

## 2. 模块介绍
`smart-gateway` 是基于 Spring Cloud Gateway 构建的统一流量网关。作为整个系统的唯一入口，它承担着路由转发、全局鉴权、多维限流、熔断降级等核心职责。其根本作用是隔离外部流量与内部下游服务，保障后端服务的高可用、安全性。

## 3. 整体架构

整体采用 **“双层限流漏斗 + 统一鉴权 + 策略化降级”** 的架构：

- **第一层（外层粗粒度）**：基于 **Sentinel** 进行路由级的总 QPS 限流与慢调用/异常比例熔断，作为系统的第一道总闸门，拦截洪峰。
- **第二层（内层细粒度）**：基于 **Redis + Lua** 实现用户/IP 级的精准令牌桶限流。同时自研了**轻量级分桶滑动窗口熔断器**，当 Redis 故障时，零延迟平滑降级到 **Guava** 本地限流。
- **全局拦截**：基于 JWT 的全局鉴权过滤器，完成身份校验与请求头清洗。
- **统一降级调度**：基于策略模式（Strategy Pattern）的 `FallbackRegistry`，针对不同接口返回定制化的 JSON 降级响应。

## 4. 技术选型及原因

| 技术组件                        | 选型原因（粗略描述）                                                          |
|-----------------------------|---------------------------------------------------------------------|
| **Spring Cloud Gateway**    | 基于 WebFlux + Netty 响应式模型，非阻塞 I/O，天然适合高并发网关场景。                       |
| **Sentinel (外层)**           | 纯 JVM 内存计算，微秒级响应，无外部依赖；具备熔断降级、系统自适应等高级治理能力，适合做粗粒度、高并发的入口总防护。        |
| **Redis (内层)**              | 支持分布式环境下的全局精准计数。解决 Sentinel 单机模式多实例部署下细粒度（用户级）限流失效的问题，且复用现有中间件，成本低。 |
| **Guava + Caffeine (内层降级)** | Redis 宕机时的本地兜底方案。Guava 提供单机令牌桶，Caffeine 管理实例防 OOM，确保网关核心防护能力永不失效。   |

## 5. 核心功能
### 1. 外层：Sentinel 限流与熔断降级
#### 1.1 设计目的
作为系统的第一道防线，保护整体后端微服务不被突发洪峰打垮，对精度要求低，但对性能和稳定性要求极高。
#### 1.2 限流熔断实现与规则设置
以路由为粒度进行限流，对 QPS 进行全局控制，对异常比例/慢调用比例进行熔断降级。

限流与熔断是Sentinel已实现的功能，我们只需配置对应规则即可。

限流规则说明：

| 参数                     | 含义     | 说明                                      |
|------------------------|--------|-----------------------------------------|
| `resource`             | 资源名    | **必须与 YAML 中的路由 `id` 完全一致**             |
| `resourceMode`         | 资源模式   | `0` = 按路由 ID 匹配（默认）；`1` = 按自定义 API 分组匹配 |
| `grade`                | 阈值类型   | `1` = QPS 限流（网关场景几乎都用 QPS）              |
| `count`                | QPS 阈值 | 单机每秒允许的最大请求数                            |
| `intervalSec`          | 统计窗口   | `1` = 按秒统计（默认）                          |
| `controlBehavior`      | 流控效果   | `0` = 直接拒绝（默认）；`1` = Warm Up；`2` = 排队等待 |
| `burst`                | 突发容量   | 类似令牌桶的 burst，允许瞬间多处理的请求数                |
| `maxQueueingTimeoutMs` | 最大排队时间 | 仅 `controlBehavior=2` 时生效               |

熔断规则说明：

| 参数                   | 含义      | 说明                                       |
|----------------------|---------|------------------------------------------|
| `resource`           | 资源名     | 与路由 ID 一致                                |
| `grade`              | 熔断策略类型  | `0` = 慢调用比例；`2` = 异常比例；`3` = 异常数         |
| `count`              | 阈值      | grade=0 时为 RT 阈值(ms)；grade=2 时为异常比例(0~1) |
| `timeWindow`         | 熔断持续时间  | 触发熔断后，服务进入 Half-Open 状态的持续秒数             |
| `minRequestAmount`   | 最小请求数   | 统计周期内至少要有这么多请求，才会触发熔断判断（防误判）             |
| `statIntervalMs`     | 统计窗口    | 统计时间窗口（毫秒），`1000` = 1秒                   |
| `slowRatioThreshold` | 慢调用比例阈值 | 仅 grade=0 时生效，慢调用占比超过此值则触发熔断             |

#### 1.3 降级 Fallback 实现
##### 1. 设计思路
- **痛点分析**：Sentinel 网关限流/熔断时，默认返回 `"Blocked by Sentinel:xxx"`，对前端极不友好；且全局统一的 fallback 无法满足复杂业务（如：秒杀、下单、查询需要不同的提示语和 HTTP 状态码）。

- 演进路径：

    - *V1.0 (硬编码)*：在 `handleRequest` 里写大量 `if-else` 判断路由和 Method。**缺点**：代码臃肿，违背开闭原则，每次加新路由都要改核心配置类。
    - *V2.0 (策略模式)*：引入策略接口和注册表，手动 `register`。**缺点**：仍需手动管理集合，容易遗漏，且顺序控制不直观。
    - *V3.0 (Spring 自动装配 + 抽象基类 - 最终版)*：利用 Spring IoC 自动收集策略，利用 `@Order` 控制优先级，利用抽象基类收敛匹配逻辑。**优点**：完美解耦，新增业务只需加一个类，零修改核心代码。

##### 2. 设计原理 

- **Sentinel 回调机制**：通过 `GatewayCallbackManager.setBlockHandler()` 劫持 Sentinel 的默认拦截行为，将控制权交给我们自定义的 `BlockRequestHandler`。
- **Spring Cloud Gateway 上下文**：利用 `ServerWebExchange` 获取当前请求的“三维坐标”：**路由ID (RouteId)**、**请求路径 (Path)**、**请求方式 (Method)**。
- **策略模式 (Strategy Pattern)**：将"匹配逻辑"与"执行逻辑"封装到独立的策略对象中，使它们可以相互替换。
- **Spring IoC 集合注入**：Spring 容器在启动时，会自动将所有实现了 `FallbackStrategy` 接口的 Bean 注入到 `List<FallbackStrategy>` 中。
- **Spring `@Order` 排序机制**：Spring 在注入 List 时，会根据 Bean 上的 `@Order` 注解（或实现 `Ordered` 接口）自动进行升序排列，解决策略匹配的优先级问题。

##### 3. 设计要点

1. **开闭原则 (OCP)**：对扩展开放，对修改关闭。新增降级场景，**绝对不允许**修改 `SentinelGatewayConfig` 和 `FallbackRegistry`，只能新增策略类。
2. **顺序敏感性**：在责任链/注册表模式中，**越精确的规则必须越先执行**。因此，精确到 POST 方法的秒杀策略 `@Order(1)` 必须排在不限方法的普通订单策略 `@Order(2)` 前面。
3. **防御性编程**：网关处于流量最前线，获取 `Route`、`Method` 时必须做非空判断，防止因极端异常导致 NPE（空指针），进而引发网关雪崩。
4. **单一职责 (SRP)**：`CommonFallback` 只负责构建 HTTP 响应；`AbstractFallbackStrategy` 只负责通用匹配逻辑；具体策略类只负责定义自己的业务逻辑。

##### 4. 类与接口职责说明

| 类/接口名                          | 类型    | 核心职责                                                                                       |
|--------------------------------|-------|--------------------------------------------------------------------------------------------|
| **`FallbackStrategy`**         | 接口    | **契约定义**。定义所有降级策略必须实现的方法：`match()` 判断是否匹配，`handle()` 执行降级响应。                               |
| **`AbstractFallbackStrategy`** | 抽象类   | **逻辑收敛**。封装通用的路由、路径、Method 匹配算法。子类只需通过构造器传入匹配条件，无需重复编写 `if-else`。                          |
| **`CommonFallback`**           | 工具类   | **响应构建**。提供静态方法，快速构建符合团队规范的 JSON 格式 `ServerResponse`。                                      |
| **`FallbackRegistry`**         | 核心组件  | **策略调度器**。利用 Spring 自动装配收集所有策略，并提供 `execute()` 方法遍历策略链，找到第一个匹配的执行。                         |
| **`SentinelGatewayConfig`**    | 配置类   | **流量入口**。拦截 Sentinel 的 Block 事件，提取请求上下文（RouteId, Path, Method），并委托给 `FallbackRegistry` 处理。 |
| **`SeckillOrderFallback`** 等   | 具体实现类 | **业务落地**。继承抽象基类，指定自己的匹配条件（如 seckill + POST），并定义具体的降级提示语。                                   |

##### 5. 设计模式使用解析
1. **策略模式**

| 角色                         | 对应代码                                     | 职责                              |
|----------------------------|------------------------------------------|---------------------------------|
| **抽象策略（Strategy）**         | `FallbackStrategy` 接口                    | 定义统一的 `match()` 和 `handle()` 契约 |
| **具体策略（ConcreteStrategy）** | `SeckillOrderFallback`、`OrderFallback` 等 | 实现具体的匹配条件和降级响应文案                |
| **上下文（Context）**           | `FallbackRegistry`                       | 持有所有策略的集合，负责调度执行                |

**利用效果**：

- **消除 if-else**：原本需要在 `handleRequest` 中写大量 `if (routeId.equals("seckill")) {...} else if (...) {...}`，现在每个分支变成了一个独立的策略类。
- **运行时动态选择**：程序在运行时根据请求的 routeId、path、method 动态决定执行哪个策略，而不是编译时写死。

2. **模板方法模式**
   
`AbstractFallbackStrategy` 就是模板方法模式的落地实现：

```java
// 父类定义了匹配的"骨架"（算法模板）
@Override
public boolean match(String routeId, String path, String method) {
    // 步骤1：校验 routeId（固定算法）
    if (StringUtils.hasText(this.routeIdPattern) && !routeId.contains(this.routeIdPattern)) {
        return false;
    }
    // 步骤2：校验 path（固定算法）
    if (StringUtils.hasText(this.pathPattern) && !path.contains(this.pathPattern)) {
        return false;
    }
    // 步骤3：校验 method（固定算法）
    return this.method == null || this.method.equalsIgnoreCase(method);
}
```

**利用效果**：

- **子类不需要关心"怎么匹配"**：`SeckillOrderFallback` 只需要在构造器中传入 `"seckill"`, `"/order"`, `"POST"` 三个参数，父类的 `match()` 方法会自动用这三个参数去做匹配。
- **防止子类写错匹配逻辑**：如果没有抽象基类，每个子类都要自己写 `match()`，很可能有人忘记判断 method，有人大小写没统一。模板方法将所有匹配逻辑收敛到一处，**保证了一致性和正确性**。

3. **责任链模式**

`FallbackRegistry.execute()` 方法就是一个**短路型责任链**（找到第一个匹配的就停止）：

```java
public Mono<ServerResponse> execute(String routeId, String path, String method, Throwable throwable) {
    for (FallbackStrategy strategy : strategies) {  // 遍历链条
        if (strategy.match(routeId, path, method)) {
            return strategy.handle(throwable);      // 匹配成功，短路返回
        }
    }
    return CommonFallback.gatewayBlockHandler();     // 链条末尾的兜底
}
```

通过 @Order 控制链条顺序，精确规则优先、宽泛规则靠后，避免大范围规则提前拦截导致精确策略失效。

#### 6. 执行流程

```text
[外部请求到达网关]
       │
       ▼
[Sentinel 拦截器匹配路由资源名]
       │
       ├─▶ 检查 QPS 是否超限 (FlowRule) ──(超限)──▶ 触发 FlowException
       │
       ├─▶ 检查是否处于熔断态 (DegradeRule) ──(熔断)──▶ 触发 DegradeException
       │
       └─▶ (通过) ──▶ 放行至内层 Redis 限流
       
[触发异常 Block]
       │
       ▼
[SentinelGatewayHandler 捕获异常]
       │
       ▼
[FallbackRegistry 遍历策略链匹配 RouteId/Path/Method]
       │
       ▼
[执行具体 FallbackStrategy (如 SeckillOrderFallback)]
       │
       ▼
[返回定制化 JSON 响应 (429/503) 给前端]
```

#### 7. 后续可优化与拓展之处

- **动态规则推送与持久化**：目前规则基于本地 JSON 文件，项目运行时无法动态更新规则。后续可以考虑以下方案：

 - 将规则存储在 Redis 中，并利用 Redis 的发布订阅机制实现动态规则推送。
 - 引入Nacos，将规则自动存储在配置中心中，并利用配置中心实现动态规则推送。

当初为什么没用配置中心？

主要是因为项目归根到底还是一个单体项目，所以单单为了一个网关中的Sentinel就引入这么一个重型组件，而且仅仅只是使用其中的规则推送和持久化，其他的服务发现、服务注册等功能均无用，还要增加维护成本，感觉没有必要。

而对于使用Redis则是没有想到，可能是由于对Redis的功能了解不全面。

### 2. 内层：Redis 令牌桶限流与 Guava 降级 (`CompositeRateLimiter`)

#### 1. 设计目的
实现用户级/IP 级的全局精准防刷，并解决 Redis 单点故障导致的限流失效和网关阻塞问题。

#### 2. 设计思路
采用**装饰器模式**实现 `RateLimiter` 接口，包装原生 `RedisRateLimiter`，对 Gateway 零侵入。内部维护一个轻量级无锁状态机，正常态走 Redis，异常态零延迟平滑降级到 Guava。

#### 3. 设计要点
1. **分桶滑动窗口**：将 10s 窗口切分为 10 个 1s 的桶（环形数组），解决固定窗口边界突刺，基于“失败率+最小样本量”决策，杜绝低流量误判。
2. **纯 CAS 无锁化**：桶过期重置摒弃 `synchronized`，使用 `AtomicReferenceArray.compareAndSet` 自旋，避免高并发线程阻塞。
3. **双重 CAS 防惊群**：探测周期内，通过双重 CAS 确保只有一个请求去探测 Redis，防止探测流量压垮刚恢复的 Redis。
4. **超时防死锁**：探测链路加入 `.timeout(2s)`，防止 Redis 连接池耗尽导致请求永久 Pending 锁死探测标志。
5. **状态恢复重置指标**：`resetToClosed()` 时强制清空滑动窗口，防止旧的高失败率数据导致系统刚恢复就瞬间再次熔断（反复横跳）。
6. **防 OOM 与 防误杀**：Caffeine 缓存 Guava 实例（上限 10 万）防恶意刷接口导致 OOM；使用突发型 `RateLimiter.create()` 确保降级瞬间提供全量 QPS。

#### 4. 执行流程

```text
[请求进入 isAllowed()]
       │
       ▼
 记录总请求数 (addTotal)
       │
       ▼
 当前状态是 OPEN 吗？ ──(是)──▶ 是否满足双重CAS探测条件？ 
       │                              │
      (否)                           (是) ──▶ [执行 Redis 探测 (带2s超时)] ─┬─(成功)─▶ 状态切 CLOSED，清空窗口
       │                              │                                    └─(失败)─▶ 保持 OPEN，降级 Guava
       ▼                              │
 [正常走 Redis 限流]                 (否) ──▶ [直接降级 Guava 本地限流]
       │
       ├─(成功)─▶ 返回结果
       │
       └─(失败/空)─▶ 记录失败 ─▶ 获取窗口快照 ─▶ 总请求>=10 且 失败率>=50%？ 
                                                │
                                               (是) ──▶ CAS 切换状态为 OPEN
                                                │
                                               (否) ──▶ 降级 Guava 本地限流
```

#### 5. 后续可优化与拓展之处

- **多维度组合限流**：目前 Key 解析主要依赖单一维度（UserID 或 IP）。后续可拓展为 `userId + path` 或 `IP + 接口` 组合键，实现更精细的接口级用户频控。
- **监控告警可视化**：将滑动窗口的失败率、熔断状态切换事件以及 Guava 降级触发次数接入 Prometheus + Grafana，实现网关容灾状态的实时可视化与飞书/钉钉告警。

### 3. 一些为什么

#### 1. 为什么要分层限流，而不是选择其中一个做全部？

- **性能与成本**：外层流量最大，若全用 Redis，每个请求都要走网络 IO + Lua 脚本，高并发下 Redis 会成为瓶颈。分层后，外层用本地内存拦截洪峰，大幅降低 Redis 压力。
- **稳定性与故障隔离**：不能把所有防线绑在同一个组件上。若全用 Redis，Redis 一宕机所有限流同时失效；分层设计天然双层生效，Redis 宕机时外层 Sentinel 依然完好，哪怕 Redis 宕机接口限流依然完好，而非直接裸奔。
- **能力边界**：Sentinel 具备熔断降级、系统自适应、热点参数识别等高级治理能力，这是纯限流器（Redis）做不到的。

#### 2. 为什么选择 Sentinel 做外层？

- 外层是系统第一道总闸门，核心诉求是**拦洪峰、保可用**，对"分布式绝对精准"要求低（差 10% 不影响稳定性）。
- Sentinel 纯 JVM 内存计算，微秒级响应，无外部依赖，即使 Redis 挂了，也不受影响。

#### 3. 为什么选择 Redis 令牌桶做内层？
- 内层核心诉求是**全局统一计数**，比如"每个用户每秒最多调用 10 次秒杀接口"，这个规则必须在所有网关实例上都生效，不能因为部署了 3 台网关，用户就能每秒调用 30 次。
- **Redis 令牌桶**：所有网关实例共享同一个 Redis 里的令牌桶，同一个用户的请求不管落到哪台机器，都扣同一个桶的配额，阈值 100% 精准，和部署实例数量无关，可以满足业务需求。
- **Sentinel 单机模式**：每台网关各自维护独立的计数器，多实例部署下，细粒度限流规则直接失效（3台机器限10次，实际能过30次）；而 Sentinel 集群模式（Token Server）虽然能解决精度问题，但部署运维成本远高于直接用 Redis，对于我的项目来说性价比极低。
- 使用 Redis 令牌桶可以实现任意维度的限流，多维度组合也只是拼接字符串的事，灵活度高，可以适配更复杂的业务规则。而 Sentinel 虽然也支持参数限流，但配置和使用复杂度高很多，高基数维度下内存占用飙升，远不如 Redis Key 灵活轻量。
除此之外，内层的流量已经是经过外层限流后的结果，即流量已被削峰，Redis 的性能压力大幅降低。

#### 4. 为什么选择 Guava 做降级限流？
- Redis 强依赖网络可用性。若 Redis 宕机或连接池耗尽，内层限流会失效甚至因等待超时拖垮网关 Netty 线程。Guava 纯内存本地令牌桶作为最后的兜底，最大程度的保障网关服务的可用性。

那内层降级到 Guava 限流时，Guava 也是单机的，那用户级防刷不就失效了吗？

当 Redis 宕机时，如果不降级到 Guava，网关要么直接报错 500（影响正常用户），要么等待超时导致 Netty 线程池耗尽（网关雪崩）。降级到 Guava，虽然变成了"单机防刷"（精度下降，比如原来限制 10 次，现在 3 台机器最多能刷 30 次），但**至少保住了系统不崩溃，且依然能挡住大部分的低级脚本刷单**。
换句话说，都要饿死了，还挑什么山珍海味。

#### 5. 为什么没有使用 Resilience4j 做熔断？
- Spring Cloud Gateway 底层是 Netty + WebFlux。引入 `resilience4j-reactor` 会在 Mono 链路上包裹重型拦截器，增加对象分配和上下文切换开销。手写的滑动窗口熔断降级仅用几个 `Atomic` 原子类在内存中维护状态，利用 Reactor 的 `onErrorResume` 进行流式切换，零外部依赖，直接跑在 Netty 线程上，性能损耗几乎为零。所以在网关 SPI 扩展点，就因地制宜选择了轻量级手写实现。

#### 6. 为什么不直接使用 Sentinel 进行熔断呢？
- 还是离不开那个问题，如果用 Sentinel 做内层限流/熔断，需要动态生成海量资源名（如 seckill-order-route:user_123），Sentinel 的资源元数据会爆炸，导致严重的内存泄漏和 GC 停顿。
- 除此之外，如果将 Sentinel 塞进内层响应式链路（包裹在 Redis Mono 链路中），Sentinel 的拦截和异常抛出会破坏 Reactor 的非阻塞语义。当 Redis 超时导致请求 Pending 时，Sentinel 无法感知底层 IO 阻塞，无法及时切断故障源，最终导致 Netty EventLoop 被大量挂起的 Redis 请求拖死。

### 3. 全局鉴权过滤器 (`AuthGlobalFilter`)

#### 1. 设计思路
网关作为统一入口，必须拦截所有非白名单请求，校验 JWT 合法性，并将可信的用户身份注入 Header，供下游微服务直接使用，避免下游重复校验。

#### 2. 设计要点

- **动态密钥与双端隔离**：通过路径前缀（`/admin/` vs `/user/`）动态选择 Admin 或 User 的 JWT 密钥和 Token Header 名称。
- **防伪造清洗**：在鉴权成功后，**强制移除**请求头中的 `Authorization`、`X-User-Id`、`X-Admin-Id`，然后再注入网关签发的真实 ID。彻底杜绝恶意客户端伪造 Header 绕过下游服务鉴权。
- **优先级控制**：`getOrder() = -1`，确保鉴权在 Redis 限流之前执行。防止恶意请求打满 Redis 令牌桶，浪费限流资源。

#### 3. 执行流程

```text
[外部请求到达网关]
       │
       ▼
[匹配请求路径]
       │
       ├─▶ 命中白名单 (登录/Swagger/公共查询) ──▶ [直接放行至下一过滤器]
       │
       └─▶ 非白名单接口 ──▶ [提取 Token]
                                  │
                                  ├─ 优先读取自定义 Header (如 "Authorization-Token")
                                  └─ 降级读取标准 Header ("Authorization: Bearer")
                                  │
                                  ▼
                           [解析 JWT 令牌]
                                  │
                           ┌─(解析失败/过期/伪造)─▶ [返回 401 JSON 响应]
                           │
                        (解析成功)
                           │
                           ▼
               [重构 ServerHttpRequest]
                   ├─ 清洗敏感 Header (防客户端伪造)
                   └─ 注入可信 ID (X-User-Id / X-Admin-Id)
                           │
                           ▼
               [放行至下一过滤器 (限流/路由)]
```

#### 4. 后续可优化与拓展之处

- **黑名单机制**：结合 Redis 实现 Token 黑名单（如用户修改密码或主动登出后，使未过期的 JWT 立即失效）。
- **RBAC 权限校验**：在网关层不仅校验身份，还可解析 JWT 中的角色信息，进行粗粒度的路由级权限拦截。

## 6. 整个网关执行流程

```text
[外部客户端发起请求]
       │
       ▼
┌─────────────────────────────────────────┐
│ 1. 全局鉴权 (AuthGlobalFilter)          │
│    - 校验 JWT，清洗并注入可信用户 ID     │
└─────────────────────────────────────────┘
       │
       ├─(鉴权失败)─▶ [返回 401 Unauthorized JSON]
       │
    (鉴权通过)
       │
       ▼
┌─────────────────────────────────────────┐
│ 2. 外层 Sentinel 防护 (路由级总闸门)    │
│    - 检查 QPS 限流 / 慢调用 / 异常熔断  │
└─────────────────────────────────────────┘
       │
       ├─(触发 Block)─▶ [SentinelGatewayHandler 拦截]
       │                     │
       │                     ▼
       │              [FallbackRegistry 策略匹配]
       │                     │
       │                     ▼
       │              [返回定制化 JSON (429/503)]
       │
    (正常放行)
       │
       ▼
┌─────────────────────────────────────────┐
│ 3. 内层 Key 解析 (RateLimiterConfig)    │
│    - 提取 Admin/User ID 或真实 IP 生成 Key│
└─────────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────────┐
│ 4. 内层复合限流 (CompositeRateLimiter)  │
│    - 正常态: Redis Lua 令牌桶扣减       │
│    - 熔断态: 降级 Guava 本地令牌桶      │
└─────────────────────────────────────────┘
       │
       ├─(令牌不足)─▶ [返回 429 Too Many Requests]
       │
    (获取令牌)
       │
       ▼
[路由转发至下游微服务 (如 localhost:8081)]
```
